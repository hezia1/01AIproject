# MVP 路线图

## 1. 总体策略

先完成一个能跑通闭环的最小平台：

项目创建 -> 仓库接入 -> 扫描任务 -> SCA/SAST 结果 -> AI 复核 -> 漏洞治理 -> 报告导出。

不要在 MVP 阶段投入重型图谱、自动 PoC、完整 DAST 或复杂多 Agent 编排。那些能力需要在稳定的数据模型和任务系统之上扩展。

## 2. 阶段一：项目骨架与平台底座

目标：

- 建立前后端工程。
- 建立数据库模型。
- 实现用户、项目、扫描任务的基础 API。

交付：

- FastAPI 后端。
- React 前端。
- PostgreSQL 数据库迁移。
- 项目列表、项目详情、扫描任务列表页面。

验收：

- 可以创建项目。
- 可以查看项目详情。
- 可以创建一条扫描任务记录。

## 3. 阶段二：仓库接入与任务执行

目标：

- 支持 Git 仓库拉取。
- 支持异步扫描任务。
- 保存扫描日志。

交付：

- Worker 进程。
- Redis 队列。
- 仓库拉取模块。
- 任务状态更新。

验收：

- 用户触发扫描后，任务状态从 queued -> running -> completed 或 failed。
- 能记录 commit hash。
- 能保存扫描日志。

## 4. 阶段三：SCA 供应链扫描

目标：

- 解析常见依赖文件。
- 生成组件清单。
- 标记许可证风险。
- 预留 CVE 数据接入接口。

交付：

- package.json 解析。
- requirements.txt 解析。
- pom.xml 解析。
- go.mod 解析。
- 组件列表页面。

验收：

- 能扫描至少一个 Node/Python/Java/Go 项目。
- 能展示依赖组件名称、版本、生态和许可证。

## 5. 阶段四：基础 SAST 规则扫描

目标：

- 实现基础规则引擎。
- 支持多语言文本级扫描。
- 输出结构化 finding。

交付：

- 规则定义格式。
- 硬编码密钥检测。
- SQL 注入危险调用检测。
- 命令执行危险调用检测。
- SSRF 危险调用检测。
- 漏洞列表页面。

验收：

- 能在示例项目中发现预置风险。
- finding 包含文件、行号、证据、严重等级和规则 ID。

## 6. 阶段五：AI 复核与修复建议

目标：

- 引入统一 AI 网关。
- 对高风险 finding 生成解释和修复建议。
- 输出误报可能性。

交付：

- AI review API。
- Prompt 模板。
- AI 复核结果表。
- 前端展示 AI 摘要。

验收：

- 对单个 finding 可以触发 AI 复核。
- AI 输出包括风险解释、影响范围、修复建议、误报可能性。

## 7. 阶段六：治理闭环与报告

目标：

- 完成漏洞状态流转。
- 支持复测记录。
- 支持报告导出。

交付：

- 漏洞状态管理。
- 备注与复测记录。
- HTML 报告。

验收：

- 一条漏洞可以完成确认、修复中、待复测、已关闭流程。
- 能导出项目级报告。

## 8. MVP 完成后的二期任务

- Agent/MCP/插件安全扫描。
- 静态发现联动动态验证。
- 沙箱行为观测。
- 误报经验库。
- 规则管理控制台。
- 多 Agent 审计编排。
- CI/CD 安全门禁。
- 行业规则包。
